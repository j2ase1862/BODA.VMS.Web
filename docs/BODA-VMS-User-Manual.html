BODA VMS 사용자 매뉴얼

버전 1.1.0 · 2026-05-22 · Vision Management System + Web MES 통합

1. 시스템 개요

BODA VMS 는 산업 현장의 머신비전 검사를 실시간 수행(VMS 클라이언트)하고, 검사 결과 · 작업지시 · 작업자 · 알람을 중앙에서 관리(BODA.VMS.Web)하는 통합 시스템입니다. 두 솔루션은 익명 REST + SignalR 로 단방향/양방향 통신합니다.

1.1 시스템 구성

항목VMS 클라이언트BODA.VMS.Web (서버)
플랫폼.NET 8 WPF (Windows 10/11 x64).NET 8 Blazor WASM + ASP.NET Core
역할실시간 비전 검사 실행, 카메라/PLC 직접 제어MES (작업지시/SPC/OEE/감사/리포트), 단일 진실 원본
레시피다운로드만 (자동 동기화)마스터 보유, CRUD 가능
검사 결과임시 메모리 + 디스크 큐 → Web 업로드영구 저장 (SQLite) + 통계 + 알람
인증로컬 시스템 사용자 BCrypt + Operator 세션 (Web 인증)JWT (관리자 / 작업자)
DB로컬 SQLite (시스템 사용자만)SQLite C:\ProgramData\BODA\VMS\BodaVision.db
외부 의존OpenCvSharp4, HelixToolkit.Wpf.SharpDX, 카메라 SDK, PLC 드라이버없음 (Kestrel 단독)

1.2 운영 흐름 한눈에

  1. VMS 실행 → 카메라 자동 연결 + Web 에 Heartbeat 시작 (5초 주기).
  2. 작업자가 헤더 Operator 로그인 (사번 + PIN). Web 의 작업자 마스터와 대조.
  3. 로그인 성공 → 헤더 Operator / Lead / Supervisor 뱃지 표시 + Work Orders 버튼 활성화.
  4. Work Orders 클릭 → Web 에서 현재 라인의 작업지시 목록 fetch → 다이얼로그.
  5. WO 선택 → RecipeName 자동 로드 + 활성 Lot 자동 채움 + AUTO RUN 활성화.
  6. AUTO RUN 클릭 → 검사 시작. 결과는 헤더 WO 칩에 진행률 실시간 표시 (예: 5/100 + ProgressBar).
  7. 계획 수량 도달 → 완료 다이얼로그 자동 표시 (KPI 카드 + 알람 사운드) + AUTO RUN 자동 정지 + 다음 WO 선택 다이얼로그.
  8. 모든 검사 1건마다 Web 의 InspectionHistory + ParameterMeasurement 에 저장. NG 발생 시 AlarmEvent 자동 생성 + 관리자 화면에 SignalR 즉시 푸시.

1.3 핵심 용어

용어설명
WO (Work Order)작업지시. 제품 × 수량 × 레시피 단위. Planned → InProgress → Completed → Closed.
LotWO 산하의 단위 묶음 (배치). 한 WO 에 여러 Lot 가능.
Operator현장 작업자. 사번 + PIN 로 키오스크 로그인. Role: Operator / Lead / Supervisor.
Recipe (Inspection Item)검사 설정 묶음. 비전 도구 + 임계값 + 파라미터.
ParamCode / ParamValue레시피 안의 개별 파라미터 (예: 패턴 점수 임계치 = 0.85).
Ctx (Context)검사 결과에 첨부하는 추적성 4필드: WO / Lot / Operator / Serial Number.
AUTO RUNVMS 의 메인 검사 시작 버튼. 운영 조건 모두 만족해야 활성.
HeartbeatVMS 가 5초마다 Web 에 보내는 생존 신호. 멈추면 5분 후 stale 세션 정리.
NG / OK판정 결과. NG 발생 시 자동 알람 생성.

2. 설치 / 배포

2.1 VMS 클라이언트 (MSI)

자체 .NET 8 런타임을 포함한 self-contained MSI (~220MB) 한 개로 설치 완료. .NET 8 별도 설치 불필요.

설치

  1. BODA-VMS-1.1.0.msi 더블 클릭.
  2. Welcome → License 동의 → Install Folder (기본 C:\Program Files\BODA VMS\) → Install.
  3. 완료 시 시작 메뉴 + 데스크톱 바로가기 자동 생성.

제거

  • 제어판 → 프로그램 및 기능 → "BODA VMS" → 제거
  • 또는 시작 메뉴의 "Uninstall BODA VMS" 바로가기

업그레이드 (v1.1 → v1.2)

새 MSI 더블 클릭하면 이전 버전을 자동 제거 후 새 버전 설치 (UpgradeCode 동일 유지). 로컬 설정 파일과 카메라 보정 등은 %LocalAppData% 에 저장되어 보존됩니다.

요구사항: Windows 10/11 x64. Visual C++ Redistributable (OpenCvSharp 네이티브 DLL 의존). 일반 Windows 에 보통 설치되어 있으나 누락 시 Microsoft 공식 사이트에서 vc_redist.x64.exe 다운로드 후 설치.

2.2 Web 서버 (PowerShell)

Kestrel 단독 + Windows Service. IIS 불필요.

설치

# 관리자 PowerShell 에서
cd D:\Project\BODA.VMS.Web\deploy
.\Install-Web.ps1

기본값: 설치 폴더 C:\Program Files\BODA VMS Web\, 포트 7144 (HTTPS), 서비스 이름 BODA-VMS-Web, 부팅 시 자동 시작 (실패 시 5초 후 자동 재시작 3회).

옵션 변경:

.\Install-Web.ps1 -Port 8443 -InstallPath "D:\BODA Web"

제거

.\Uninstall-Web.ps1

SQLite DB 는 보존 ($env:LOCALAPPDATA\BODA\VMS). 완전 삭제 시:

Remove-Item "$env:LOCALAPPDATA\BODA\VMS" -Recurse

2.3 네트워크 / 방화벽

측포트 / 프로토콜방향용도
Web 서버7144 / HTTPS인바운드VMS → Web REST API + SignalR (자동 방화벽 룰 생성)
Web 서버5000-5100 / HTTP(선택)VisionServer 통합 시
VMS 클라이언트—아웃바운드Web 서버 7144 접속, 카메라 GigE 3956 (UDP)
HTTPS 인증서: 개발 환경은 self-signed. 첫 실행 시 dotnet dev-certs https --trust. 운영은 사설 CA 또는 HTTP.

2.4 첫 실행 전 체크리스트

  1. Web 서버 먼저 시작 → 브라우저로 https://<서버IP>:7144 접속 확인.
  2. 관리자 계정으로 회원가입 → 승인 (현재 첫 사용자가 자동 Admin).
  3. Operators 페이지에서 작업자 마스터 등록 (사번 + PIN + Role).
  4. Products / Recipes / Work Orders 1건 이상 등록.
  5. VMS 클라이언트 설치 후 첫 실행 → AppSetup 마법사에서 WebServerUrl = https://<서버IP>:7144, ClientIndex = 1, 2, 3... 라인별 지정.
  6. VMS 실행 → 시작 화면 헤더에 Heartbeat 정상 표시 확인.

3. VMS 클라이언트 매뉴얼

3.1 첫 실행 화면

실행 시:

  1. Splash → 카메라 자동 검색 (백그라운드).
  2. 로컬 시스템 사용자 로그인 (엔지니어용, BCrypt). 선택
  3. 메인 윈도우 표시: 헤더 + KPI Strip + 카메라 캔버스 + 시스템 로그 + 사이드 패널 (접힘).

3.2 헤더 — 운영 흐름

좌→우로 자연스러운 운영 동선. 모든 칩 높이 통일 (32px), AUTO RUN 만 강조 (34px).

3.2.1 Operator 로그인

청록 보더 칩. 로그인 전 "Login..." 버튼만 표시.

  1. "Login..." 클릭 → 다크 다이얼로그 (사번 + PIN).
  2. Web /api/kiosk/login 으로 대조.
  3. 성공 시 헤더 칩 형태가 "김기동 (1003) Lead [Logout]" 으로 변경.

3.2.2 Work Orders 버튼

Operator 로그인 후 활성. 클릭 시 Web 에서 작업지시 목록 fetch → 980×640 chromeless 다이얼로그.

  • 상태 필터: 활성(Planned + InProgress) / 전체 / Planned / InProgress / Completed.
  • 컬럼: Order No / Product / Recipe / Progress (예: 5/100) / Status (컬러 뱃지) / Planned Start.
  • 더블 클릭 또는 Select 버튼으로 선택.

3.2.3 WO 칩 (진행률 ProgressBar)

선택된 작업지시 표시. 검사 1건마다 실시간 갱신.

WO: WO-20260522-001 · A001 (5/100)  [▰▰▱▱▱▱▱▱]
  • ProgressBar 색: 청록 (#14B8A6). 트랙: #1A1A1C.
  • PlannedQuantity = 0 인 WO 는 ProgressBar 숨김.
  • 완료(Status=Completed) 시 자동 unselect + 다음 WO 선택 다이얼로그.

3.2.4 Recipe 칩

파란 보더 (#3D7DE8) 책 아이콘 + 레시피명. WO 선택 시 WO.RecipeName 으로 로컬 레시피 자동 로드 → 카메라 전파 → Web 파라미터 동기화. 로컬에 없으면 Web 동기화 후 재시도.

3.2.5 Ctx (Inspection Context)

업로드 시 자동 첨부될 추적성 4필드:

필드채움 방식비고
WOWO 선택 시 자동수동 오버라이드 가능
LotWO 선택 시 활성 Lot 자동 fetch활성 Lot 없으면 비움
S/N바코드 스캐너 수동 입력제품 시리얼 — 검사 1회 1S/N
Op IDOperator 로그인 시 자동로그아웃 시 비움

✕ 버튼으로 모든 필드 클리어.

3.2.6 AUTO RUN

운영 흐름의 단일 Primary 액션. 모든 조건 만족 시 청록 (#00ADB5) 활성, 실행 중에는 빨강 STOP 으로 전환.

3.2.7 Role 뱃지

Operator chip 안에 권한 뱃지:

  • Operator — 기본, 뱃지 없음
  • Lead — 반장. 파란 뱃지.
  • Supervisor — 관리자. 주황 뱃지.

3.3 사이드 패널 (Settings)

헤더 우측 톱니바퀴 아이콘 (⚙) 으로 토글. 320px 폭.

Camera Control 섹션

  • Grab — 단일 이미지 캡처. AUTO RUN 비활성 시만 활성.
  • Live Start / Live Stop — 연속 라이브 그랩.
  • Roller — 롤러 검사 모드 토글 (지원 카메라만).

Recipe 섹션

  • 현재 로드된 레시피 + 버전.
  • Available Recipes 리스트 (로컬 + Web 동기화).
  • Load / New / Save / Export / Import 버튼. New/Save/Delete 는 권한 필요.

External Tools 섹션 Supervisor 전용

  • Vision Tool Setup — VMS.VisionSetup 엔지니어 워크스페이스 실행 (드래그로 비전 도구 파이프라인 구성).
  • System Setup — VMS.AppSetup 마법사 (카메라 / PLC / 네트워크 / 사용자).

Operator / Lead 등급은 이 섹션 자체가 숨겨집니다.

Web Parameters 섹션

  • Sync Parameters — Web 파라미터 즉시 동기화 (60초 자동 주기 외 수동 트리거).

Recent Inspections 섹션 (D8)

VMS 자체 In-Memory 순환 버퍼 (최대 200건). Web 끊겨도 작업자가 즉시 확인.

  • 3-cell 통계: Total / Pass / NG + Pass rate.
  • ListBox 최근 N건 — 시간 (HH:mm:ss) + OK / NG 뱃지 + 시리얼 · WO · Recipe + NG 코드.
  • Clear 버튼으로 초기화. ToolTip 으로 풀 타임스탬프.
프로세스 재시작 시 비워짐. 영구 저장은 Web InspectionHistory 가 담당.

Statistics 섹션

세션 누적 통계: Total / Pass / Fail / Pass Rate.

3.4 작업지시 완료 다이얼로그

계획 수량 도달 시 자동 표시 (560×500 chromeless 다크, 청록 2px 보더):

  • 큰 원형 ✓ 아이콘 (72px) + WO OrderNo (22pt Bold).
  • KPI 카드 4개: 계획 / 검사 / Pass / NG (각각 색상 차별화).
  • Pass rate · 진척률 표시.
  • 표시 시 SystemSounds.Asterisk 자동 재생 (작업자 주의 환기).
  • 하단 두 버튼: 닫기 / 다음 작업지시 선택 (Enter 기본).

3.5 AUTO RUN 활성화 조건

다음을 모두 만족해야 활성:

  1. 카메라 1대 이상 연결 (HasConnectedCamera).
  2. 시스템 사용자 StartStop 권한 (UserPermission.StartStop) — 미로그인 시 기본 허용.
  3. (Web 통합 환경) Operator 로그인 상태.
  4. (Web 통합 환경) WO 선택 상태이고 Status 가 Planned 또는 InProgress.

비활성 시 ToolTip 으로 사유 표시. Standalone (Web 미연결) 환경은 4번 항목 우회.

4. BODA.VMS.Web (관리자 / MES)

브라우저로 접속 (https://<서버IP>:7144). MudBlazor 기반 다크 테마. 사이드 메뉴 + 상단 AppBar (다국어 전환 / 알림 / 사용자).

4.1 로그인 / 회원가입

  1. "/login" 페이지에서 사용자명 + 비밀번호 입력.
  2. 회원가입 시 첫 사용자는 자동 Admin, 이후는 Pending 상태 → 기존 Admin 이 승인 필요.
  3. JWT 토큰 발급 → localStorage 저장 → 모든 API 호출에 자동 첨부.

4.3 Dashboard

실시간 라인 KPI: 전체 라인 카운트, Pass Rate, NG 비율, 활성 작업지시, 최근 알람 위젯. SignalR 로 실시간 갱신.

4.4 Andon Board

전체 라인 실시간 상태 그리드. 각 라인 카드에 클라이언트명 + 현재 작업자 + WO 진행률 + 알람 카운트. 알람 카운트 클릭 시 Alarms 페이지로 이동 (필터 적용).

4.5 Alarms

알람 목록 + 상세. 필터: 라인 / 상태 (New/Ack/Resolved) / 심각도 / 기간.

액션

  • Acknowledge (확인) — 작업자가 인지함. 배지 카운트에서 빠짐.
  • Resolve (해제) — 다이얼로그에서 사유 입력 후 해제.
  • 새 알람은 SignalR AlarmCreated 로 즉시 push.

4.6 Production History

검사 이력 (InspectionHistory + ParameterMeasurement). 필터: 라인 / 기간 / Pass/NG / WO / Operator.

  • 각 행 클릭 시 NgDetailDialog 팝업 — 측정값 + 사진 + NG 코드.
  • Excel 내보내기.
  • 읽기 전용 (감사 추적성 보존).

4.7 Work Orders

작업지시 마스터. Planned → InProgress → Completed → Closed.

  • + 새 WO 다이얼로그 — Product / Client (라인) / Recipe / Planned Quantity / Planned Start.
  • OrderNo 자동 채번 (WO-{YYYYMMDD}-{NNN}).
  • Start 버튼 — Planned → InProgress 수동 전이 (또는 VMS 첫 결과 도착 시 자동).
  • Complete — VMS 가 자동 (PlannedQuantity 도달) 또는 수동.
  • Close — Completed 상태에서 영구 마감.

4.8 Products

제품 마스터 (코드 / 이름 / 카테고리 / 활성 상태). WO 의 ProductId 가 참조.

4.9 Inspection Items (Recipes + Parameters)

레시피와 그 산하 파라미터 관리. VMS 가 60초 주기로 자동 동기화.

  • 레시피 트리: Pattern / Blob / Dimension 카테고리.
  • 파라미터: ParamCode (정수) / ParamValue (소수) / Description / Unit / LowerLimit / UpperLimit / IsActive.
  • VMS 측에서 검사 시 ParamValue 사용. LowerLimit/UpperLimit 은 향후 SPC 판정용.

4.10 Operators

3개 탭:

  • Operator Master — 사번 / 이름 / 부서 / Role / 활성. PIN 등록 (BCrypt 해시).
  • Sessions — 로그인/로그아웃 이력. VMS 로그인 시 SignalR 로 실시간 갱신 — 새로 고침 불필요.
  • Kiosk Links — 각 라인의 키오스크 페이지 URL.

Role 설명

Role설명VMS 헤더 뱃지
Operator일반 작업자. 검사 실행만.없음
Lead반장. + Recipe 편집.Lead 파랑
Supervisor관리자. + External Tools (Vision Tool / System Setup).Supervisor 주황

4.11 Lots

WO 산하 단위 묶음. LotNumber 자동 채번 ({YYYYMMDD}-{OrderNo}-{Seq:D3}). Open → Closed.

VMS 가 WO 선택 시 활성(Open) Lot 1개 자동 채움. 활성 Lot 없으면 Lot 없이 진행 가능.

4.12 Maintenance

예방정비 (PM) 스케줄. Overdue / DueToday / Upcoming. Perform 다이얼로그로 수행 기록.

4.13 Defect Codes

NG 코드 마스터 (코드 / 설명 / 카테고리 / 심각도 Critical/Major/Minor).

4.14 Shifts

교대 정의 (시작 시간 + 시간 길이). 검사 결과의 ShiftId 자동 매핑.

4.15 Audit Logs

모든 마스터 데이터 변경 이력 (Create / Update / Delete). 사용자 / IP / 시각 기록. 읽기 전용.

4.16 Reports / Quality Analysis

OEE 지표, 라인별 / 작업자별 / WO별 통계, SPC 차트 (LCL/UCL).

4.17 알림 (NotificationBell)

상단 AppBar 우측 종 아이콘. 배지로 미확인 알람 수 표시 (100+ 시 "99+").

액션

  • 아이콘 클릭 → 최근 10건 + "N건 미확인" 헤더 + "모두 확인" 버튼.
  • 각 알람 클릭 → Alarms 페이지로 이동.
  • 각 알람 우측 ✕ 아이콘 → 개별 확인 처리.
  • 모두 확인 버튼 → 모든 New 알람 일괄 Acknowledge.
  • 모두 감사 추적성을 위해 삭제가 아닌 Acknowledge — 알람 이력 영구 보존.

5.2 추적성 4필드

5.3 업로드 큐 (오프라인 복구)

Web 끊김 / 재시작 시 검사 결과 손실 방지.

  • 업로드 실패 → %LocalAppData%\BODA VISION AI\upload_queue\{UTC timestamp}_{guid}.json 디스크 보존.
  • 5초 주기 백그라운드 timer 가 큐 디렉토리 스캔 → 시간 순 정렬 → 한 건씩 재전송.
  • 한 건 실패 시 즉시 중단 (순서 보존 + 서버 다운 시 스팸 방지).
  • VMS 프로세스 재시작 후에도 큐 파일은 디스크에 남아 자동 복구.
  • 손상된 JSON 은 자동 삭제 (무한 재시도 방지).

5.4 SignalR 실시간 푸시

두 개의 SignalR Hub:

Hub경로인증이벤트
VmsHub/hubs/vmsJWT 필수AlarmCreated / AlarmUpdated / NgAlert / ClientStatusChanged / OperatorSessionStarted / OperatorSessionEnded
VmsPublicHub/hubs/vms-public익명WorkOrderUpdated / WorkOrderCompleted (VMS 클라이언트용)

VMS 가 한 검사를 업로드하면:

  1. 본인은 응답 본문에서 진행률 즉시 받음 (1차).
  2. Web 이 동시에 모든 SignalR 연결에 WorkOrderUpdated broadcast (2차).
  3. 같은 라인의 다른 VMS (있다면), 관리자 Dashboard / Andon / Alarms 페이지 모두 새로고침 없이 즉시 반영.

5.5 다중 라인 운영

여러 VMS 가 같은 Web 서버에 연결 가능. 각 VMS 는 ClientIndex 로 구분 (1, 2, 3...).

  • 각 라인의 WO 는 /api/workorders/by-client/{clientIndex} 로 분리.
  • 각 라인의 Operator 세션도 ClientId 별로 독립.
  • 관리자 Andon Board 에서 전체 라인 동시 모니터링.
ClientIndex 유일성: 동일 ClientIndex 로 두 VMS 인스턴스 실행 금지. Web 이 IP 덮어쓰기 + 경고만 함. 운영 규칙으로 라인별 ClientIndex 를 사전 배정.

7. 트러블슈팅

7.1 VMS 측

실행 시 DLL 오류

Visual C++ Redistributable 누락. Microsoft 사이트에서 vc_redist.x64.exe 설치.

BrushBgWindow 리소스 찾을 수 없음

VMS Launcher 가 VMS.VisionSetup 다이얼로그를 띄울 때 발생. 이미 EnsureWindowStylesMerged() 로 자동 처리되지만, 다른 다이얼로그 추가 시 같은 패턴 사용 필요.

카메라 자동 연결 실패

  • System Setup 의 카메라 IP / 매뉴팩처러 확인.
  • 방화벽: GigE Vision 은 UDP 3956 사용.
  • 각 NIC 마다 broadcast 보내므로 NIC 메트릭 확인.

업로드 큐가 계속 쌓임

Web 연결 끊김. %LocalAppData%\BODA VISION AI\upload_queue\ 에 .json 누적. Web 서버 시작 후 5초 내 자동 드레인. 직접 확인은 파일 수 모니터링.

AUTO RUN 이 활성 안 됨

다음 모두 확인:

  1. 카메라 1대 이상 연결 됨 (헤더에 연결 표시)
  2. Operator 로그인 됨 (Operator chip 작업자 이름 표시)
  3. WO 선택 됨 (WO chip 표시)
  4. WO Status 가 Planned 또는 InProgress (Completed 면 비활성)

버튼에 마우스 올리면 ToolTip 으로 사유 확인 가능.

7.2 Web 측

Windows Service 가 시작 안 됨

  • Event Viewer → Windows Logs → Application → "BODA-VMS-Web" 로그.
  • 주요 원인: 포트 충돌 (netstat -ano | findstr 7144), HTTPS 인증서 미신뢰.
  • HTTPS 인증서: dotnet dev-certs https --trust.

1003 민병준 같은 작업자가 계속 작업중으로 표시

VMS 비정상 종료로 OperatorSession 이 active 로 남음. 자동 처리:

  • VMS graceful 종료 시 /api/clients/disconnect 에서 세션 자동 종료.
  • Web 시작 시 5분 이상 heartbeat 없는 클라이언트의 active 세션 일괄 Stale 정리.
  • 수동 정리: DB 에서 직접 UPDATE 또는 Web 재시작.

Operators Sessions 탭이 갱신 안 됨

v1.1 부터 SignalR 로 자동 갱신. 만약 안 되면:

  • 브라우저 콘솔 (F12) → Network → WS 탭 → /hubs/vms 연결 상태 확인.
  • JWT 만료 시 재로그인.

알림 배지 안 보임

NotificationBell 의 CSS vms-notif-badge 가 적용되도록 브라우저 캐시 클리어 (Ctrl+Shift+R).

7.3 통합 / 네트워크

VMS 가 Web 에 연결 못 함

  1. VMS System Setup → WebServerUrl 확인 (예: https://192.168.0.10:7144).
  2. Web 서버 PC 의 방화벽: Get-NetFirewallRule -DisplayName "BODA VMS Web*" 확인.
  3. self-signed 인증서: 운영은 사설 CA 인증서 사용 권장. 임시 우회는 VMS 의 HttpClient 가 cert validation 우회 (dev only).

다른 VMS 의 검사가 내 VMS 헤더에 반영 안 됨

SignalR 연결 끊김. VMS 콘솔 출력 확인: [VmsHubClient] Connected → https://.../hubs/vms-public. 자동 재연결 정책 (0s/2s/10s/30s 무한) 으로 보통 자체 복구.

검사 결과가 Web 에 안 보임

  1. VMS 업로드 큐 (upload_queue\) 에 .json 쌓여 있는지 확인 — 있으면 네트워크 끊김.
  2. Web Event Viewer 로그에서 /api/parameters/results 호출 확인.
  3. Recipe ID / Client Index 가 양쪽에서 일치하는지 확인.
